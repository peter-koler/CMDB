# Feature Specification: Frontend Vben UI Refactor

**Feature Branch**: `[006-vben-ui-refactor]`  
**Created**: 2026-03-25  
**Status**: Draft  
**Input**: User description: "参考 vue-vben-admin 深度重构前端代码，重构现有 frontend 应用为纯白商务风格、固定视口滚动布局，保留现有权限/路由/API/生命周期逻辑，并支持明暗主题切换。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 重构全局布局与设计系统 (Priority: P1)

作为监控平台用户，我希望进入系统后看到统一的白色商务化布局、固定头部与侧边栏、仅内容区滚动的页面骨架，这样我可以更稳定地浏览系统并获得一致的视觉体验。

**Why this priority**: 这是所有页面的共同基础。如果布局骨架、主题 token、菜单和头部不先统一，后续页面级重构会重复返工且视觉无法收敛。

**Independent Test**: 登录后访问仪表盘、CMDB、监控管理、系统设置等任意一级页面，确认头部和侧边栏固定、仅内容区滚动、菜单权限与高亮正常、通知与用户菜单可用、明暗主题可以切换。

**Acceptance Scenarios**:

1. **Given** 用户已登录并进入系统主界面，**When** 页面完成渲染，**Then** 布局必须采用固定视口结构，侧边栏和头部不随主内容滚动。
2. **Given** 用户在任意业务页面切换路由，**When** 路由变化后，**Then** 菜单高亮、展开项、面包屑与当前页面路径保持一致。
3. **Given** 用户具备不同权限组合，**When** 主菜单渲染时，**Then** 所有现有权限判断和菜单可见性必须与重构前保持一致。
4. **Given** 用户点击主题切换入口，**When** 在明亮与深色主题间切换，**Then** 系统必须切换到对应主题且页面结构、功能和可读性保持正常。

---

### User Story 2 - 统一核心业务页面样式 (Priority: P2)

作为平台用户，我希望列表页、详情页、配置页和仪表盘在卡片、筛选区、表格、表单和状态信息上采用统一规范，这样我可以在不同模块之间获得一致的操作体验。

**Why this priority**: 布局骨架稳定后，用户最常接触的是仪表盘、列表和配置页面，统一这些页面可以最快体现重构价值。

**Independent Test**: 打开仪表盘、通知中心、个人资料、授权页及若干 CMDB/监控/系统页面，确认卡片、标题、筛选栏、表格与表单风格统一，且页面功能未回退。

**Acceptance Scenarios**:

1. **Given** 用户访问列表页， **When** 页面加载完成，**Then** 筛选区、操作栏、表格、分页与空状态必须遵循统一的间距、边框和配色规范。
2. **Given** 用户访问详情页或配置页，**When** 页面渲染完成，**Then** 表单、描述信息、辅助说明和操作区必须遵循统一的容器和层级规范。
3. **Given** 用户访问仪表盘，**When** 指标卡和图表面板渲染完成，**Then** 页面必须保留原有数据内容与交互入口，同时呈现统一的商务化视觉风格。

---

### User Story 3 - 完成长尾页面收口与回归验证 (Priority: P3)

作为维护者，我希望剩余页面、暗色主题、响应式细节和通用组件都完成统一收口，并通过构建与类型检查验证，这样这次重构可以被后续会话继续推进和安全交付。

**Why this priority**: 长尾页面和最终验证决定交付质量，也是这次大型重构能否持续推进的边界条件。

**Independent Test**: 完成长尾页面覆盖后执行类型检查、构建和关键路径人工回归，确认系统可运行且没有明显样式断裂或 TS 报错。

**Acceptance Scenarios**:

1. **Given** 重构覆盖长尾页面后，**When** 用户遍历主要业务入口，**Then** 不应出现明显的视觉割裂、滚动冲突或样式回退。
2. **Given** 开发者执行前端类型检查与构建，**When** 命令完成，**Then** 不应出现新的 TypeScript 报错或构建失败。
3. **Given** 用户使用移动端或窄视口访问系统，**When** 页面自适应调整，**Then** 主要导航、操作区与内容区仍可正常使用。

### Edge Cases

- 当用户权限较少时，菜单项减少后仍必须保持正确的选中态、展开态和面包屑路径。
- 当自定义视图数量较多或视图名称较长时，侧边栏折叠与展开状态下都不能破坏菜单布局。
- 当通知数量较多或通知中心打开时，固定头部与内容区滚动不能互相干扰。
- 当页面内容很长、表格很多或图表较大时，只允许内容区滚动，不允许整个应用容器产生双滚动条。
- 当切换到深色主题时，边框、文字、交互态和图表容器必须维持足够对比度，不能因为保留暗色兼容而出现不可读内容。
- 当页面存在 `v-if` 权限判断、`onMounted` 数据加载、WebSocket 初始化和路由监听时，重构不得改变这些逻辑的触发时机。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须为现有 `frontend/` 应用建立统一的全局设计系统，覆盖主色、背景色、边框、圆角、间距、文字层级和状态色。
- **FR-002**: 系统必须将主应用布局重构为固定视口结构，保证头部与侧边栏固定，仅内容区滚动。
- **FR-003**: 系统必须保留并继续显示现有完整菜单体系，包括 CMDB、监控管理、告警中心、系统设置及动态自定义视图菜单。
- **FR-004**: 系统必须保留现有 `hasPermission`、`hasAnyPermission` 及其所有调用行为，不得因重构改变权限可见性。
- **FR-005**: 系统必须保留现有站点配置加载、自定义视图加载、通知初始化、WebSocket 连接和路由监听逻辑。
- **FR-006**: 系统必须在头部提供动态面包屑，并与当前路由状态同步。
- **FR-007**: 系统必须保留通知角标、通知中心、语言切换、主题切换和用户下拉菜单，并以统一商务化风格重绘。
- **FR-008**: 系统必须支持明亮主题与深色主题切换，并保证两套主题都能正常使用。
- **FR-009**: 系统必须把列表页、详情页、配置页、表单页和仪表盘收敛到统一的容器、卡片、筛选栏、表格和操作区样式规范。
- **FR-010**: 系统必须优先通过全局样式变量和通用页面样式完成重构，减少逐页散落定义，方便后续统一调整。
- **FR-011**: 系统必须避免修改后端代码、接口契约、业务数据结构以及 `manage-go`、`collector-go` 等非前端代码。
- **FR-012**: 系统必须保证重构后前端代码无新增 TypeScript 报错，并可完成正常运行所需的构建校验。
- **FR-013**: 系统必须允许将本次重构拆分为可中断、可恢复的阶段性任务，并在文档中记录每阶段的范围与验收方式。

### Key Entities *(include if feature involves data)*

- **Layout Shell**: 应用级骨架，包含侧边栏、头部、内容区、固定视口行为、折叠态和滚动规则。
- **Theme Tokens**: 全局视觉变量，定义亮色/暗色主题下的颜色、边框、阴影替代、间距与组件状态。
- **Navigation State**: 菜单选中项、展开项、面包屑和路由之间的同步状态。
- **Page Surface**: 页面容器、卡片容器、工具栏、筛选栏、表格区、表单区和信息区等通用视觉承载层。
- **Protected UI Logic**: 不允许改动的权限判断、API 请求、生命周期调用、通知初始化与主题状态逻辑。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户在任意主页面首次进入后，能在 5 秒内识别固定头部、固定侧边栏和可滚动内容区的布局结构。
- **SC-002**: 至少覆盖主布局、仪表盘、通知中心、个人资料、授权页以及主要业务页面容器，使主要页面的视觉规范一致性达到可复用状态。
- **SC-003**: 100% 现有一级菜单分组、子菜单项和动态自定义视图在重构后继续可访问，且权限行为与重构前一致。
- **SC-004**: 明亮与深色主题切换后，关键路径页面均能保持文本与交互可读，无阻断性显示问题。
- **SC-005**: 执行前端类型检查与生产构建时，不出现新的 TypeScript 报错或构建失败。
- **SC-006**: 在桌面端和常见窄视口场景下，系统不出现布局级双滚动条、头部错位或侧边栏遮挡内容的明显问题。
