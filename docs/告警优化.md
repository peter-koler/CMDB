# 告警优化（冻结版需求与开发计划）

> 更新时间：2026-03-13  
> 目标：在现有 Redis 采集完成基础上，补齐 HertzBeat 风格的“实时告警 + 周期告警 + 模板默认告警 + 实例级覆写”能力，面向商用版本。

---

## 1. 当前代码基线（已确认）

### 1.1 已有能力
1. `manager-go` 已有告警引擎与周期评估框架：
   - 实时表达式引擎：`manager-go/internal/alert/engine.go`
   - 周期评估器（基于 VM Query）：`manager-go/internal/alert/periodic.go`
   - 告警事件存储（内存）：`manager-go/internal/alert/event_store.go`
2. `python-web` 已有告警规则管理/通知规则/告警中心接口与页面：
   - 路由：`backend/app/routes/monitoring_target.py`
   - 前端：`frontend/src/views/monitoring/alert/*.vue`
3. 现有规则模型（`alert_defines`）已具备 `type/expr/period/times/template/notice_rule_id` 等字段，可承载实时与周期规则。

### 1.2 关键缺口
1. `manager-go` 当前实时告警仅示例规则（`manager_dispatch_tick_high`），未按监控模板/实例配置动态加载规则。
2. 实时告警变量输入当前仅是单点变量（`value` 和 `${metrics}_${field}`），无法直接支持 Redis 场景中的跨字段比值/状态判断（如 `used_memory / maxmemory`、`rdb_last_bgsave_status != ok`）。
3. 告警规则管理目前主要在 Python Web 侧 CRUD，`manager-go` 未形成“规则热加载 + 按 monitor 绑定 + 在线评估”闭环。
4. 监控详情页尚无“实例告警 Tab（模板默认 + 实例覆写 + 当前状态）”。

### 1.3 数据契约约束（新增冻结）
1. 告警相关库表已存在，`manager-go` 必须按现有共享表对齐，不再新增平行表结构。
2. 本阶段以 `docs/migration-plan-new.md` 作为 DB 契约来源，重点对齐：
   - `alert_defines`
   - `single_alerts`
   - `alert_history`
   - `notice_rules`
   - `notice_receivers`
   - `notice_templates`
3. 允许新增索引与必要兼容字段映射，但不改动既有业务语义和 Python Web 侧模型定义。

## 2. 需求冻结（V1）

### 2.1 范围（In Scope）
1. **模板默认告警**：在监控模板（先 Redis）定义默认告警项，并可导入到规则库。
2. **实例级告警配置**：监控实例可继承模板默认规则，并允许覆写：
   - 告警级别
   - 阈值/表达式
   - 触发次数与周期
   - 通知规则
   - 消息模板
   - 启停状态
3. **实时告警能力补齐**：`manager-go` 在采集入库路径对实时规则进行在线评估并触发。
4. **周期告警保留并统一**：沿用 VM Query 周期告警，规则来源统一到共享规则模型。
5. **告警数量控制**：Redis 默认启用“核心规则集”，其余规则默认关闭但可一键启用。

### 2.2 暂不纳入（Out of Scope）
1. 跨应用（非 Redis）完整默认规则包一次性上线。
2. 复杂多窗口统计 DSL（如小时级 slowlog 聚合）第一版仅提供可落地近似方案。
3. 告警分组/抑制/静默机制重写（沿用现有能力，后续增强）。

## 3. Redis 默认告警规则（冻结清单）

> 采用“核心 8 条默认启用 + 扩展 8 条默认关闭”策略，降低噪声。

### 3.1 核心默认启用（8 条）
1. 实例不可用（实时）
2. 内存使用率过高（周期）
3. 内存碎片严重（周期）
4. 连接数饱和（实时）
5. 拒绝连接（实时）
6. RDB 失败（实时）
7. AOF 失败（实时）
8. 主从延迟过高（周期）

### 3.2 扩展默认关闭（8 条）
1. 内存使用率预警
2. 内存碎片预警
3. 命中率突降
4. 命中率异常
5. 主从延迟预警
6. 从节点离线
7. 阻塞客户端过多
8. 慢查询过多

## 4. 设计方案（V1）

### 4.1 规则来源与绑定
1. 复用 `alert_defines` 作为规则主表，不新造平行规则表（强约束）。
2. 新增/复用“规则-监控绑定”关系（按 monitor_id），支持：
   - `source=template_default | instance_override | user_custom`
   - `enabled`、`severity`、`notice_rule_id`、`template` 覆写
3. 模板导入时生成模板默认规则快照（与 `template_id` 关联），实例创建时可一键应用。

### 4.2 实时评估引擎增强（manager-go）
1. 新增 monitor 维度的“最近指标快照缓存”（短 TTL，建议 2-5 分钟）。
2. 单个 metric point 到达时，更新快照并触发该 monitor 的实时规则评估。
3. 表达式变量支持从快照取值（不再仅单点），满足比值/状态类规则。
4. 字符串状态判断（如 `!= "ok"`）V1 通过映射变量处理：
   - 例如 `rdb_last_bgsave_status_ok=1/0`，表达式使用数值比较。

### 4.3 周期评估统一
1. 周期规则统一从规则表加载。
2. 周期调度按规则 `period` 执行，继续基于 VM Query。
3. 与实时规则共用告警去重/降噪链路（reducer + queue + dead letter）。

### 4.4 前端交互（监控详情页新增告警 Tab）
1. 展示实例当前绑定规则列表（来源、启停、阈值、级别、通知规则、消息模板）。
2. 支持实例级编辑与恢复默认。
3. 展示该实例“当前告警/最近历史”摘要。

## 5. API 变更（冻结）

### 5.1 Python Web（统一出口）
1. `GET /monitoring/targets/{id}/alerts/rules`：获取实例规则（含来源与覆写信息）。
2. `PUT /monitoring/targets/{id}/alerts/rules/{rule_id}`：更新实例规则覆写。
3. `POST /monitoring/templates/{template_id}/alerts/apply`：将模板默认规则应用到实例（批量）。
4. `POST /monitoring/targets/{id}/alerts/reload`：触发 manager 规则热加载（可选）。

### 5.2 Manager-Go（内部）
1. 新增规则加载器（从共享 DB 拉取 enabled 规则 + 绑定关系）。
2. 新增实时规则评估入口（collector report ingest 路径）。
3. 新增规则热加载接口（供 Python Web 调用，或定时 reload）。

## 6. 分阶段开发计划

### Phase A：规则运行时闭环（后端优先）
1. `manager-go` 实现规则加载器与缓存（直接读取共享 DB 的 `alert_defines`）。
2. 打通实时规则评估（按 monitor 绑定）和周期规则统一加载。
3. 告警落库对齐共享表：
   - 触发写 `single_alerts`
   - 关闭/恢复写 `alert_history`
4. 增加规则热加载与基础观测日志。

**验收**：Redis 单实例可触发“实例不可用/连接饱和/RDB失败”等实时规则。

### Phase B：模板默认规则与实例覆写
1. 模板默认规则入库机制（先 Redis）。
2. 实例创建时可选择“应用模板默认告警”。
3. 实例规则覆写接口与持久化。

**验收**：同一模板下不同实例可有不同阈值/通知策略。

### Phase C：前端告警 Tab 与运营化
1. 监控详情新增告警 Tab。
2. 支持启停、阈值、级别、通知规则、消息模板编辑。
3. 展示当前告警摘要与最近变更记录。

**验收**：无需切到“全局告警设置”即可完成实例级告警运维。

## 7. 风险与对策

1. 规则过多导致告警风暴：  
   - 默认仅启用核心规则；提供“推荐规则”分层。
2. 实时表达式变量缺失导致误报：  
   - 增加变量缺失审计日志与规则健康状态。
3. Manager 重启/热更导致规则漂移：  
   - 规则版本号 + 热加载日志 + reload 失败回退。
4. 与 HertzBeat 语义差异：  
   - 规则清单与表达式语义做兼容映射表并回归测试。

---

## 8. 冻结结论

本文件作为告警优化 V1 的冻结需求基线，按“先后端闭环，再模板绑定，再前端增强”的顺序实施。  
冻结后不再扩展新协议/新应用规则，优先把 Redis 告警能力做到可商用上线。

## 11. 告警升级实施记录（更新于 2026-03-14）

### 11.1 本次实现范围
1. 新增告警规则升级配置字段：`alert_defines.escalation_config`。
2. Python Web 规则接口已支持升级配置透传与校验：
   - 读取：规则列表/详情返回 `escalation_config`。
   - 写入：创建/更新实例规则支持 `escalation_config`。
   - 模板默认规则应用时可写入升级配置。
3. manager-go 已支持升级配置热加载：
   - `alert_runtime_store` 在加载规则时读取 `escalation_config`。
   - 对旧库无该字段场景做了列存在性兼容（不会因为缺列导致 Manager 启动失败）。
4. manager-go 新增告警升级 Worker（Redis ZSET 延时队列）：
   - 队列 Key：`alert:escalation:queue`
   - Stage Key：`alert:escalation:stage:{group_key}`
   - 默认每 10 秒扫描一次到期任务。
   - 使用 Lua 脚本原子领取到期任务，避免多实例重复发送。
5. 升级触发与终止逻辑：
   - 首轮告警进入新周期（new cycle）后按 Level1 等待时间入队。
   - 执行升级前反查 `single_alerts`：仅 `status=firing` 继续。
   - 若告警已恢复，或已被人工认领/确认（labels 中存在 assignee/ack 标记）则终止升级并清队列。
   - 每级通知发送后按下一级等待时间继续排期，直至最高级。
6. 升级记录落库：
   - 每次升级通知发送后，写入 `alert_history` 一条 `status='firing'` 记录，内容为升级通知文案（用于时间线展示）。

### 11.2 升级配置格式（已实现）
```json
{
  "enabled": true,
  "levels": [
    {
      "level": 1,
      "delay_seconds": 300,
      "notice_rule_ids": [101, 102],
      "title_template": "[Level {{level}}] {{rule_name}}",
      "content_template": "告警 {{rule_name}} 在 {{instance}} 持续未恢复"
    }
  ]
}
```
说明：
- `delay_seconds` 必须大于 0。
- `notice_rule_ids` 支持数组；也兼容单值 `notice_rule_id`。
- 若某级未配置 `notice_rule_ids`，运行时会回退到规则自身 `notice_rule_ids`。
