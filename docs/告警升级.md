# 监控平台：告警升级功能开发任务说明书 (Technical Spec)
## 1. 任务背景
[cite_start]当前系统是基于 **Go (Manager) + Python (Flask) + Vue 3** 构建的监控平台 [cite: 30, 31, 32, 33][cite_start]。为了满足企业级运维需求，需要在现有告警架构（Calculator -> Reducer -> Notifier）中增加**告警升级 (Alarm Escalation)** 功能 [cite: 52, 54, 55]。
**性能目标**：支持 10,000+ 监控对象，处理每分钟数千条告警，要求高并发、低延迟且不阻塞主流程。
业务逻辑：告警触发后，若在指定时间内状态仍为“待处理”，则按照配置的阶梯（Level）自动发送通知给更高级别的接收人
---
## 2. 业务逻辑流 (Business Logic)
1. [cite_start]**触发与入队**：告警触发并完成首轮（Level 0）通知后 [cite: 40, 169]，检查是否有升级策略。若有，计算下一级触发时间并存入 Redis ZSET。
2. **状态机检查**：升级 Worker 定时扫描。执行升级前必须反查告警状态：
   - 若状态为 `待处理 (Firing)`：执行当前级别的通知，并排期下一级升级。
   - [cite_start]若状态为 `处理中 (Acknowledged)` 或 `已恢复 (Resolved)` [cite: 57, 130]：立即终止升级流程，从队列移除。
3. **阶梯通知**：按照 `Level 1 -> Level 2 -> Level N` 的顺序依次向上级负责人发送通知。

## 4. 高性能存储设计 (Redis)
[cite_start]禁止使用数据库轮询，必须复用现有 Redis 缓存层 [cite: 34, 167]。
1. **延时队列 (ZSET)**：
   - **Key**: `alert:escalation:queue`
   - [cite_start]**Member**: `{group_key}` (对应告警的分组键) [cite: 90, 137]。
   - **Score**: `Next_Trigger_Timestamp` (下次执行升级的绝对时间戳)。
2. **阶段跟踪 (Hash)**：
   - **Key**: `alert:escalation:stage:{group_key}`
   - **Field**: `current_level` (记录当前处于第几个阶梯)。

## 5. 后端逻辑实现 (Go Manager)
[cite_start]**涉及目录**：`manager-go/internal/alert/` [cite: 68]
1. [cite_start]**规则热加载**：修改 `alert_runtime_store.go` [cite: 70]，确保 `escalation_config` 被加载到内存中，避免运行期查库。
2. **Escalation Worker (核心)**：
   - [cite_start]实现一个独立协程，默认是 10 秒执行 `ZRANGEBYSCORE alert:escalation:queue -inf {now}` [cite: 74]，
   - **原子判定**：使用 Lua 脚本或原子操作捞取任务，防止多实例重复发送。
   - [cite_start]**反查逻辑**：查询 `single_alerts` 表 [cite: 94] 确认状态是否仍为“未处理”。
3. [cite_start]**通知下发**：调用现有 `NotifyService` 发送对应级别的渠道通知（Webhook/邮件/企业微信等） [cite: 74, 154]。
4.若仍为“待处理”，调用 NotifyService 发送对应级别的通知 。
5.循环迭代：根据配置更新 ZSET 的 Score 为下一级时间，直到到达最高等级

## 7. 性能约束
- **异步解耦**：所有升级操作必须在独立协程中完成，不得影响核心采集流程 [cite: 82]。
- **批量回写**：升级事件产生的 `alert_history` [cite: 95] 记录需采用异步批量写入模式，保护数据库 IO。
-严禁轮询数据库：升级判定必须基于 Redis ZSET 和内存规则缓存。
