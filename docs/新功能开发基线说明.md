# 新功能开发基线说明

更新时间：2026-03-26

本文档用于约束后续新功能的前端、后端、联调与交付方式。目标不是限制开发速度，而是确保新增功能继续沿用当前仓库已经稳定下来的架构、视觉语言和接口习惯，避免新代码把系统重新拉回“页面风格分裂、接口契约不统一、联调成本高”的状态。

## 1. 当前基线结论

- 前端 UI 重构已经形成统一壳层与页面表面体系，新增页面必须默认接入该体系。
- 前端运行基线：
  - 路由壳层：`frontend/src/layouts/BasicLayout.vue`
  - 全局主题与页面表面变量：`frontend/src/assets/styles/index.css`
  - 请求封装：`frontend/src/utils/request.ts`
  - 路由定义：`frontend/src/router/index.ts`
- 后端运行基线：
  - 应用工厂：`backend/app/__init__.py`
  - 路由注册入口：`backend/app/routes/__init__.py`
  - 业务服务目录：`backend/app/services/`
  - Flask 运行入口：`backend/run.py`
- 当前开发数据库基线：
  - 元数据主库推荐使用 PostgreSQL 16
  - 推荐 `DATABASE_URL=postgresql+psycopg2://arco_user:arco_password@127.0.0.1:5432/arco_db`
  - `manager-go` 运行时存储仍有 SQLite 遗留，涉及授权、监控 manager 侧数据时需要显式考虑这一层

## 2. 新功能设计原则

### 2.1 页面和交互原则

- 新页面默认放在统一壳层内，不再自行实现独立 header、breadcrumb、侧栏风格。
- 页面默认使用 `app-page`、`app-surface-card`、`app-toolbar` 等统一类，不再新造一套页面容器。
- 新功能优先复用现有模块心智模型：
  - 列表页：筛选区 + 表格区 + 批量操作区
  - 详情页：概览卡片 + Tab 内容区
  - 配置页：表单区 + sticky 操作区
  - 图形页：普通信息与画布逻辑分层，避免把复杂交互直接塞进单个页面组件
- 所有新增页面都必须考虑窄屏和高分辨率显示，不允许只在本机默认窗口尺寸下看起来正常。

### 2.2 代码组织原则

- 前端：页面负责展示和交互编排，接口调用放入 `src/api/`，复杂状态和副作用拆到 composable 或独立 helper。
- 后端：路由层负责参数接入、鉴权、响应格式；业务规则放入 `services`；数据结构与持久化逻辑放在 `models`。
- 不在页面组件或 Flask 路由里堆积长业务逻辑。
- 新功能优先“在既有模块上延展”，不要为了一个页面新起一套平行实现。

### 2.3 一致性原则

- 接口返回保持现有形态：`{ code, message?, data? }`
- 权限控制继续沿用现有 menu permission 体系，不新增页面级“临时判断”
- 主题继续使用 CSS 变量和 Ant Design Vue token 映射，不回退到写死颜色
- 测试和验证至少覆盖：类型检查/构建/核心路径人工验证

## 3. 前端开发基线

### 3.1 技术与目录

- 技术栈：Vue 3 + TypeScript + Vite 5 + Ant Design Vue 4 + Pinia + Vue Router
- 核心目录：
  - `frontend/src/views/`：页面
  - `frontend/src/components/`：复用组件
  - `frontend/src/api/`：接口封装与类型
  - `frontend/src/stores/`：全局状态
  - `frontend/src/composables/`：页面复用逻辑
  - `frontend/src/layouts/`：布局壳层
  - `frontend/src/assets/styles/`：全局主题与样式基线

### 3.2 页面实现规范

- 新页面必须优先采用以下结构：

```vue
<template>
  <div class="app-page xxx-page">
    <div class="app-page__header">
      ...
    </div>

    <a-card :bordered="false" class="app-surface-card">
      ...
    </a-card>
  </div>
</template>
```

- 如果是列表页，优先沿用：
  - 顶部筛选表单
  - 中部表格或卡片列表
  - 底部批量操作或分页
- 如果是详情页，优先沿用：
  - 顶部摘要信息
  - `a-tabs` 分区
  - 子区块仍使用 `app-surface-card`
- 不要在新页面重新定义一整套背景、边框、阴影、字号体系。

### 3.3 路由与菜单规范

- 所有新页面都要在 `frontend/src/router/index.ts` 中完整定义：
  - `name`
  - `meta.title`
  - `meta.icon`
  - `meta.permission`
  - 若不出现在菜单中，显式声明 `hideInMenu: true` 或对应隐藏标记
- 需要菜单高亮和 breadcrumb 的页面，不要绕开路由元信息硬编码。
- 新增详情页、编辑页、隐藏页时，必须检查它在 `BasicLayout.vue` 中的 breadcrumb 和高亮是否符合当前导航体系。

### 3.4 样式与主题规范

- 新样式优先使用全局变量，不直接写死色值：
  - 文本：`--app-text-primary` / `--app-text-secondary` / `--app-text-muted`
  - 表面：`--app-surface-card` / `--app-surface-subtle`
  - 边框：`--app-border`
  - 主色：`--app-accent` / `--app-accent-soft`
- 允许保留品牌视觉、业务热力图、拓扑状态色等少量语义色，但必须经过明确说明，不能默认整页使用硬编码颜色。
- 响应式基线：
  - flex 子项需要注意 `min-width: 0`
  - 筛选表单在窄屏下要能换行或改单列
  - 表格宽度不足时优先提供横向滚动或列裁剪策略，不能直接裁掉操作区
- 图表、G6、富文本、CodeMirror 这类第三方组件，必须做主题适配或清晰界定暂不适配范围。

### 3.5 组件与状态规范

- 复杂页面状态不要全部塞在一个 `reactive` 里；列表条件、弹窗状态、详情状态尽量拆开。
- 表格列定义、表单 schema、枚举映射等可抽出常量或 helper，避免页面 script 无限膨胀。
- 重复逻辑优先沉淀到：
  - `src/components/`
  - `src/composables/`
  - 页面目录内 `helpers.ts` / `constants.ts`
- 页面中出现 300 行以上的纯业务转换逻辑时，应优先考虑拆分。

### 3.6 接口与类型规范

- 所有页面调用统一走 `frontend/src/utils/request.ts`
- 新接口定义放在对应领域文件中，例如：
  - 监控：`src/api/monitoring.ts`
  - CMDB：`src/api/cmdb.ts`
  - 通知：`src/api/notifications.ts`
- 接口文件中要同时维护请求函数和核心 TS 类型，避免页面层随处写 `any`
- 类型策略：
  - 页面收到的列表、详情、统计对象都要有显式接口
  - 第三方库缺少声明时，在 `src/types/` 增补本地声明
  - 新代码不要把 `typecheck` 问题留给后续 feature 一起承担

### 3.7 前端开发完成前最低检查

每个前端功能至少完成以下检查：

1. `cd frontend && npm run typecheck`
2. `cd frontend && npm run build`
3. 人工验证以下项目：
   - 菜单高亮与 breadcrumb
   - 权限显示与禁用态
   - 窄屏表现
   - 表格/弹窗/抽屉是否溢出
   - 亮色/暗色主题是否可读
   - 关键交互链路是否闭环

## 4. 后端开发基线

### 4.1 技术与目录

- 技术栈：Python 3.11 + Flask 3.0 + Flask-SQLAlchemy + Flask-Migrate + Flask-JWT-Extended
- 核心目录：
  - `backend/app/models/`：ORM 模型
  - `backend/app/routes/`：HTTP 路由
  - `backend/app/services/`：业务服务
  - `backend/app/notifications/`：通知与 WebSocket 相关逻辑
  - `backend/app/tasks/`：调度任务
  - `backend/migrations/`：Alembic 迁移
  - `backend/tests/`：测试

### 4.2 路由层规范

- 新接口必须挂到已有领域 blueprint 下；只有明确形成新领域时才新增 blueprint。
- 路由层只做以下几件事：
  - 参数获取与基础校验
  - `jwt_required` / 权限控制
  - 调用 service
  - 返回统一响应结构
- 不要把大段查询拼装、复杂规则计算、外部系统调用堆在路由函数里。

建议沿用当前返回风格：

```python
return jsonify({"code": 200, "message": "success", "data": payload})
```

失败时返回合适 HTTP 状态码，并同时给出稳定可读的 `message`。如果已有领域定义了 `error_code`，继续沿用，不要混用多套错误格式。

### 4.3 服务层规范

- 下列逻辑优先放入 `backend/app/services/`：
  - 规则匹配
  - 跨模型/跨表聚合
  - 与 manager-go/collector-go 等外部服务的交互
  - 复杂数据转换、默认策略拼装、回退逻辑
- service 应尽量可复用、可测试，不依赖具体 route 上下文。
- 若某个功能包含“本地数据 + manager-go 代理 + fallback”，优先参照现有监控模块的模式，而不是在多个接口里分别手写。

### 4.4 数据与迁移规范

- 新增持久化字段或表时，必须同步补迁移，不接受“先手改模型，迁移以后再补”。
- 涉及授权、监控、调度的功能，先确认数据真实归属：
  - PostgreSQL 元数据
  - manager-go 运行时
  - collector-go
  - 历史 SQLite 遗留
- 如果功能跨越 Python 后端与 Go 服务，必须先定义主写入侧，避免双写漂移。

### 4.5 权限与安全规范

- 新接口默认需要鉴权，匿名接口必须白名单说明用途。
- 菜单权限、按钮权限、接口权限要能互相对上，不能只做前端隐藏。
- 涉及文件上传、模板执行、外部回调、通知发送的功能，要显式校验输入边界和异常回退。
- 不要绕过现有 License 守卫逻辑；如新增免授权接口，需要明确加入白名单并说明原因。

### 4.6 测试规范

- 后端新增规则、聚合、导入导出或权限逻辑时，优先补：
  - service 单测
  - route 集成测试
- 当前测试入口可参考：

```bash
cd backend
pytest
```

- 至少保证新增功能的关键成功路径和一个失败路径有测试覆盖。

## 5. 前后端联动基线

### 5.1 接口契约

- 前端不要直接猜字段名，后端不要无通知地改字段结构。
- 列表接口优先明确：
  - `items`
  - `total`
  - 分页参数名
- 详情接口优先返回扁平稳定结构；只有在确实需要时再返回嵌套大对象。
- 枚举、状态值、布尔字段要在前后端保持同一语义，不混用中英文和数字状态。

### 5.2 回退与兼容

- 若接口依赖 manager-go，而 manager-go 某些能力可能未实现，应明确：
  - 正常路径
  - 404/5xx fallback
  - 前端空态表现
- 回退逻辑必须收敛到服务层或公共 helper，避免前端和后端各自猜测对方状态。

### 5.3 交付顺序

推荐按以下顺序开发新功能：

1. 明确页面入口、权限归属、数据归属
2. 先定后端契约与返回结构
3. 再做前端页面骨架与交互流
4. 再做主题、细节和响应式收口
5. 最后做类型检查、构建、测试与人工回归

## 6. 明确禁止的做法

- 前端新页面脱离 `BasicLayout` 与 `app-page` 体系单独造样式
- 页面内大面积写死颜色、边框、阴影、字号
- 只在前端做权限隐藏，不补后端接口权限
- 在路由函数中直接堆复杂业务逻辑
- 新增接口不定义 TS 类型，靠页面层 `any` 贯穿
- 有迁移需求但不补 Alembic migration
- 不跑 `typecheck/build/pytest` 就标记功能完成
- 遇到历史技术债时顺手扩大重构范围，导致 feature 交付跑偏

## 7. 新功能提测前检查清单

提交前至少逐项确认：

- 页面是否接入统一壳层、breadcrumb、菜单高亮
- 页面是否使用主题变量而不是写死视觉样式
- 页面在 1440px、窄屏下是否仍可用
- 表格、弹窗、抽屉、图表是否存在遮挡或裁切
- 接口是否落在正确领域模块
- 返回结构是否符合当前统一响应格式
- 是否补充或更新了相关类型、迁移、测试
- `frontend` 的 `npm run typecheck` 与 `npm run build` 是否通过
- `backend` 的关键测试是否通过
- 是否补充了必要文档或操作说明

## 8. 建议的文档维护方式

- 若后续只是普通功能迭代，直接以本文档为总基线，不必每次新建“工程规范”文档。
- 若后续发生以下任一变化，应同步更新本文档：
  - 前端壳层或主题体系大改
  - 权限模型调整
  - API 响应规范变化
  - PostgreSQL / manager-go / collector-go 数据边界变化
  - 构建、测试、发布流程变化
